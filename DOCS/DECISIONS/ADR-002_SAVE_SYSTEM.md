# ADR-002: 세이브 시스템 — localStorage 싱글턴 + 씬 재시작 UI 갱신

> **카테고리:** DECISIONS
> **최초 작성:** 2026-03-21
> **최종 갱신:** 2026-03-21
> **관련 기능:** SaveSystem.js, CurrencySystem.js, UpgradeScene.js

**상태:** Accepted
**날짜:** 2026-03-21

---

## 맥락

영구 세이브(Gold, 해금 유닛, 업그레이드 레벨, 클리어 스테이지)를 저장해야 했다. 두 가지 설계 결정이 필요했다.

1. 세이브 저장소로 무엇을 사용할 것인가
2. 업그레이드 구매 후 UI를 어떻게 갱신할 것인가

---

## 결정 1: localStorage 기반 세이브

`localStorage`를 세이브 저장소로 사용하며, `SaveSystem`을 IIFE 싱글턴으로 구현한다.

- 저장 키: `defense_save`
- 형식: JSON 직렬화
- 메모리 캐시(`_cache`)로 매 접근 시 localStorage 파싱 비용을 제거
- 세이브 데이터 변경 후 `persist()`를 명시적으로 호출해야 저장됨

---

## 이유 (결정 1)

- 서버 인프라 없이 영구 데이터를 저장하는 브라우저 표준 방법이다
- JSON 직렬화/역직렬화로 구조화된 데이터를 간단하게 저장한다
- 싱글턴 패턴으로 세이브 데이터가 항상 단일 인스턴스에서 관리된다 — 복수의 독립 로드로 인한 데이터 불일치를 방지한다
- `Object.assign({}, DEFAULT_SAVE, parsed)` 방식의 로드로, 게임 업데이트로 새 필드가 추가되어도 기존 세이브가 손상되지 않는다

---

## 검토한 대안 (결정 1)

**sessionStorage:**
- 탭을 닫으면 데이터가 소실된다. 영구 세이브 요건을 충족하지 못한다. 기각.

**IndexedDB:**
- 비동기 API이므로 현재 동기식 세이브 접근 패턴과 맞지 않는다
- 5MB를 초과하는 데이터가 필요하지 않으므로 도입 이점이 없다. 기각.

**Cookie:**
- 용량 제한(4KB)이 있어 게임 데이터 저장에 부적합하다. 기각.

---

## 결정 2: 업그레이드 구매 후 씬 재시작(scene.restart())

UpgradeScene에서 해금 또는 업그레이드 구매 후 `this.scene.restart()`로 씬 전체를 재시작하여 UI를 갱신한다.

---

## 이유 (결정 2)

UpgradeScene의 UI는 세이브 데이터를 기반으로 각 항목의 상태(해금 여부, 현재 레벨, 비용, 버튼 활성화)를 계산하여 렌더링한다. 구매가 발생하면 세이브 데이터가 변경되므로 연관된 여러 UI 요소가 동시에 갱신되어야 한다.

개별 요소 갱신 방식은 다음 문제를 야기한다:
- 각 행의 Graphics, Text 객체 참조를 개별적으로 추적해야 한다
- 해금 완료 후 버튼을 제거하고 "해금 완료" 텍스트로 교체하는 등 구조적 변경이 필요하다
- 코드 복잡도가 높아지고 일부 상태를 누락할 위험이 있다

씬 재시작 방식은:
- `create()`가 항상 세이브 데이터를 최신 상태로 읽어 전체 UI를 다시 그린다
- 구현 코드가 단순하다
- 상태 누락이 불가능하다
- 재시작 시간이 플레이어에게 인지될 만큼 길지 않다

---

## 결과

**긍정적 트레이드오프:**
- 세이브 로직이 단일 모듈(`SaveSystem.js`)에 집중되어 있다
- 세이브 데이터 구조 변경 시 `DEFAULT_SAVE`와 이 ADR만 갱신하면 된다
- 씬 재시작 방식으로 UpgradeScene UI 코드가 단순하게 유지된다

**부정적 트레이드오프:**
- `persist()`를 명시적으로 호출해야 하므로, 호출을 누락하면 세이브가 저장되지 않는다. 세이브 데이터를 수정하는 모든 코드는 반드시 `SaveSystem.persist()`를 호출해야 한다
- 씬 재시작 시 화면이 짧게 깜빡일 수 있다 (현재 성능상 문제없음)
- localStorage는 동일 브라우저/경로에서만 유효하므로, 다른 브라우저나 다른 파일 경로로 이동 시 세이브가 유지되지 않는다
